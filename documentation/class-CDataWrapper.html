Class CDataWrapper

Data wrapper.

This class overloads its ancestor to implement the framework for providing services that store and request data. The class does not feature a concrete data store engine, so it only concentrates in providing the framework for derived classes that implement concrete data store instances, for this reason this class is abstract.

These new functionalities require an additional set of parameters:

  • Request parameters: These parameters refer to the parameters that the service expects:
    • kAPI_SERVER: Server, this offset tag will be used to locate the service CServer object, concrete derived classes should instantiate this object in their CWrapper::_InitResources() method and store it in the $_REQUEST global array.
    • kAPI_DATABASE: Database, this string parameter represents the name of the database which is a concrete instance of CDatabase.
    • kAPI_CONTAINER: Container, this string parameter represents the name of the container within the provided database,it will be a concrete instance of CContainer.
    • kAPI_PAGE_START: Page start, this integer parameter represents the start offset for paging, it is a zero-based index.
    • kAPI_PAGE_LIMIT: Page limit, this integer parameter represents the maximum number of records to be returned.
    • kAPI_QUERY: Query, this array parameter represents the data query, it should follow the rules of the CQuery class family.
    • kAPI_SELECT: Select, this array parameter represents the query selection or fields list, it is constituted by an array of strings that indicate which fields should be returned by the query.
    • kAPI_SORT: Sort, this array parameter represents the list of fields upon which to sort the results, the array element indexes represent the field references, the values are provided as numbers where negative values represent a descending sense and zero or positive values ascending.
  • Response parameters: These parameters refer to the parameters that the service is expected to return:
    • kAPI_PAGING: Paging, this offset is constituted by the list of parameters relating to paging:
      • kAPI_PAGE_START: Page start (input).
      • kAPI_PAGE_LIMIT: Page limit (input).
      • kAPI_PAGE_COUNT: Page count (output), the actual number of elements returned, which will be less or equal to the kAPI_PAGE_LIMIT parameter.
    • kAPI_STATUS: Status, this block features an additional element that is used by query requests:
      • kAPI_AFFECTED_COUNT: Affected count (output): the total number of elements affected by the operation.
  • Operations: This class adds the following operations:
    • kAPI_OP_COUNT: Count, this operation will return the total number of elements that satisfy the provided kAPI_QUERY; if the latter is omitted, it is assumed that we want the total number of elements in the kAPI_CONTAINER. The operation expects the following parameters:
      • kAPI_FORMAT: Format, this parameter is required, since other non scalar parameters must be encoded.
      • kAPI_DATABASE: Database, database to which the container belongs.
      • kAPI_CONTAINER: Container, container in which to perform the query.
      • kAPI_QUERY: Query (optional), the selection criteria.
    • kAPI_OP_QUERY: Query, this operation will return all records that satisfy the provided query. The operation expects the following parameters:
      • kAPI_FORMAT: Format, this parameter is required, since other non scalar parameters must be encoded.
      • kAPI_DATABASE: Database, database to which the container belongs.
      • kAPI_CONTAINER: Container, container in which to perform the query.
      • kAPI_QUERY: Query (optional), the selection criteria.
      • kAPI_SELECT: Select (optional), the list of fields to be returned.
      • kAPI_SORT: Sort (optional), the list of sort elements and their sense.
      • kAPI_PAGE_LIMIT:This parameter is required or enforced, it represents the maximum number of elements that the query should return, the default value is kDEFAULT_LIMIT.

# _ParseRequest( )

Parse request.

Parse request.

This method should be used to parse the request, check the request elements and make any necessary adjustments before the request is CDataWrapper::_ValidateRequest().

This is also where the relevant request elements will be logged to the relative response sections.

The method is called by the constructor and should be overloaded to handle derived classes custom elements.

In this class we handle the following parameters:

  • kAPI_DATABASE: We copy the database name to the request.
  • kAPI_CONTAINER: We copy the container name to the request.

# _ParseOperation( )

Parse operation.

Parse operation.

We overload this method to remove unnecessary parameter from the request, depending on the current operation:

  • kAPI_OP_COUNT: This operation does not handle fields selection or sort orders, so we clear kAPI_SELECT and kAPI_SORT. We could also clear the paging parameters, but we keep them in case clients implement an automatic paging management.
  • kAPI_OP_QUERY: This operation requires that the page limits be set, only a defined number of records should be returned by a service: if the kAPI_PAGE_LIMIT parameter was not provided, this method will set it to kDEFAULT_LIMIT.
