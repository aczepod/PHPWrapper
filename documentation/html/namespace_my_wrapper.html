Default attributes.

This file contains the tag definitions for all default attributes, these terms represent the concept identifiers for all the default object properties.

  • Identification attributes: These definitions tag the identification attributes:
    • kTAG_NID: Native identifier. This attribute contains the native unique identifier of the object. This identifier is used as the default unique key for all objects and can have any scalar data type.
    • kTERM_LID: Local identifier. This attribute contains the local unique identifier. This value represents the value that uniquely identifies an object within a specific domain or namespace. It is by definition a string constituting the suffix of the global identifier, kTERM_GID.
    • kTERM_GID: Global identifier. This attribute contains the global unique identifier. This value represents the value that uniquely identifies the object across domains and namespaces. This value is by definition a string and will generally constitute the object's primary key (kTERM_NID) in full or hashed form.
    • kTERM_UID: Unique identifier. This attribute contains the hashed unique identifier of an object in which its kTERM_NID is not related to the kTERM_GID. This is generally used when the kTERM_NID is a sequence number.
    • kTERM_SYNONYMS: Synonyms. This attribute contains a list of strings that represent alternate identifiers for the hosting object. Synonyms do not have any relation to the namespace.
  • Classification attributes: These attributes are used to classify objects:
    • kTERM_CATEGORY: Category. This attribute is an enumerated set that contains all the categories to which the hosting object belongs to.
    • kTERM_KIND: Kind. This attribute is an enumerated set that defines the kind of the hosting object.
    • kTERM_TYPE: Type. This attribute is an enumerated set that contains a combination of data type and cardinality indicators which, combined, represet the data type of the hosting object.
    • kTERM_CLASS: Class. This attribute is a string that represets the referenced object's class. This attribute is used to instantiate the correct class once an object has been retrieved from its container.
    • kTERM_NAMESPACE: Namespace. This attribute is a reference to a term which represents the namespace of the current object. The object local identifiers must be unique within the scope of the namespace.
    • kTERM_INPUT: Input. This attribute is an enumerated set that defines an input type, it is generally associated to attributes and defines what kind of control should be used in a form to input a value for that attribute.
  • Description attributes: These attributes are used to describe objects:
    • kTERM_LABEL: Label. This attribute represents the label, name or short description of the referenced object. It is an kTYPE_LSTRING structure in which the label can be expressed in several languages.
    • kTERM_DEFINITION: Definition. This attribute represents the definition of the referenced object. It is an kTYPE_LSTRING structure in which the definition can be expressed in several languages. A definition is independent of the context.
    • kTERM_DESCRIPTION: Description. This attribute represents the description of the referenced object. It is an kTYPE_LSTRING structure in which the description can be expressed in several languages. A description depends on the context.
    • kTERM_NOTES: Notes. This attribute represents the notes or comments of the referenced object. It is a kTYPE_LSTRING structure in which the description can be expressed in several languages.
    • kTERM_EXAMPLES: Examples. This attribute represents the examples or templates of the referenced object. It is a list of strings where each string represents an example or template.
  • Status attributes: These attributes are used to describe a status:
    • kTERM_STATUS_LEVEL: Severity. This attribute represents the severity or status level, it is an element of the following enumerated set:
      • kSTATUS_IDLE: This code indicates no errors, this state can be equated to an idle state.
      • kSTATUS_NOTICE: This code indicates a notice. A notice is an informative message that does not imply an error, nor a situation that should be handled; it can be considered as statistical data.
      • kSTATUS_MESSAGE: This code indicates a message. A message is an informative message that is addressed to somebody, although it does not imply an error or warning, it was issued to a receiving party.
      • kSTATUS_WARNING: This code indicates a warning. Warnings are informative data that indicate a potential problem, although they do not imply an error, they indicate a potential problem or an issue that should be addressed, at least at a later stage.
      • kSTATUS_ERROR: This code indicates an error. Errors indicate that something prevented an operation from being performed, this does not necessarily mean that the whole process is halted, but that the results of an operation will not be as expected.
      • kSTATUS_FATAL: This code indicates a fatal error. Fatal errors are errors that result in stopping the whole process: in this case the error will prevent other operations from being performed and the whole process should be halted.
      • kSTATUS_BUG: This code indicates a bug. Bugs, as opposed to errors, result from internal causes independant from external factors. A bug indicates that an operation will never execute as stated, it does not necessarily mean that it is fatal, but rather that the behaviour of an operation does not correspond to its declaration.
    • kTERM_STATUS_CODE: Code. This attribute represents the status code, it is a string or number that identifies the specific status or state.
    • kTERM_STATUS_MESSAGE: Message. This attribute represents the status message, it is a string that describes the status. It is a kTYPE_LSTRING structure in which the message can be expressed in several languages.
  • Authorship attributes: These attributes are used to describe authorship:
    • kTERM_AUTHORS: Authors. This attribute represents a list of authors, it is an array of author names.
    • kTERM_ACKNOWLEDGMENTS: Acknowledgments. This attribute represents a list of generic acknowledgments, it is an array of strings.
    • kTERM_BIBLIOGRAPHY: Bibliography. This attribute represents a list of bibliographic references, it is an array of strings.
  • Reference attributes: These attributes are used to reference other objects and contain the native identifier of the referenced object:
    • kTERM_TERM: Term. This attribute contains a reference to an object that represents the term of the attribute host.
    • kTERM_NODE: Node. This attribute contains a reference to an object that represents the node of the attribute host.
    • kTERM_SUBJECT: Subject. This attribute contains a reference to an object that represents the start, origin or subject vertex of a subject/predicate/object relationship.
    • kTERM_OBJECT: Object. This attribute contains a reference to an object that represents the end, destination or object vertex of a subject/predicate/object relationship.
    • kTERM_PREDICATE: Predicate. This attribute contains a reference to an object that represents the predicate term of a subject/predicate/object relationship.
    • kTERM_PATH: Path. This attribute represents a sequence of vertex, predicate, vertex elements whose concatenation represents a path between an origin vertex and a destination vertex.
  • Reference collections: These attributes are used as collections of references to other objects and contain a list whose elements are the native identifier of the referenced object:
    • kTERM_NAMESPACE_REFS: Namespace references count. This integer attribute counts how many times external objects have referenced the current one as a namespace.
    • kTERM_NODES: Nodes. This attribute is a collection of node references, it is an array of node object native identifiers who reference the current object.
    • kTERM_EDGES: Edges. This attribute is a collection of edge references, it is an array of edge object native identifiers who reference the current object.
    • kTERM_FEATURES: Features. This attribute is a collection of feature references, it is an array of object native identifiers that reference the current object as a feature or trait.
    • kTERM_METHODS: Methods. This attribute is a collection of method references, it is an array of object native identifiers that reference the current object as a method or modifier.
    • kTERM_SCALES: Scales. This attribute is a collection of scale references, it is an array of object native identifiers that reference the current object as a scale or unit.

Default categories.

This file contains the term definitions for all default categories, these terms represent the concept identifiers for all the default object categories.

  • Main categories: These terms represent the ontology main default categories:
    • kTERM_ONTOLOGY: Ontology. An ontology is a graph structure composed of nodes related by a series of directed edges, each node represents a concept that can be used to illustrate a characteristic or feature of an object, or to tag a set of data elements that are measurements of that feature.
    • kTERM_NAMESPACES: Namespaces. Namespaces are terms which are containers for a set of term identifiers or names based on their functionality or semantic origin. Such terms are used to allow the disambiguation of homonym identifiers residing in different namespaces.
    • kTERM_PREDICATES: Predicates. Predicates are terms which are used to link a subject vertex to an object vertex into a relationship or graph directed edge.
    • kTERM_ATTRIBUTES: Attributes. Attributes or properties are the elements that comprise an object, these may represent categories, descriptions or features. In this ontology, attributes are implemented by a term, which represents the attribute definition, a node, which represents the attribute description and the tag, which represents the attribute name.
    • kTERM_ENUMERATIONS: Enumerations. Enumerations are terms which represent elements of a controlled vocabulary or enumerated set. Enumeration term local identifiers can be used in local scope, while their global identifier can unambiguously be used in any scope.
  • Attribute categories: These terms represent the main categories to which the attributes belong:
    • kTERM_IDENTIFICATION: Identification. Identification attributes are those which allow the identification, disambiguation or discrimination of objects. These attributes are used to make an object unique.
    • kTERM_REFERENCE: Reference. Reference attributes allow the identification and retrieval of related objects. This type of relationship differs from the graph kind of relationships in that the object embeds the reference to the other object and does not attach a predicate to the link.
    • kTERM_CLASSIFICATION: Classification. Classification attributes are categorical qualifications that group objects around a series of indicators or aggregators. These indicators are generally used as summary keys.
    • kTERM_ILLUSTRATION: Illustration. Illustration attributes are used to provide definition, description or qualitative information for an object.
    • kTERM_AUTHORSHIP: Authorship. Authorship attributes are used to indicate the origin, provenance or authority for an object.
    • kTERM_STATUS: Status. Status attributes indicate the state, level or outcome of an object or an event. These attributes are also used to provide statistical information.
  • Enumeration categories: These terms represent the main categories of enumerated sets:
    • kTERM_TYPES: Types. A type is a generalised qualification applied to an object. In general this kind of terms will be used to indicate a data type, scale or unit.
    • kTERM_KINDS: Kinds. A kind is similar to a type, except that while the type is general, the kind is specific to the class of the object. Kinds are used to cluster objects of the same class
    • kTERM_STATES: States. State terms indicate the status, condition or state of an object or event. These classifications are generally used to qualify the outcome of an operation.
    • kTERM_OPERATORS: Operators. Operators indicate an operation type, they require a subject operand and often relate the subject to an object operand. Operators are predicates, except that they always produce a result.
  • Type categories: These terms represent the type categories:
    • kTERM_PRIMITIVES: Primitives. Primitive types are scalar data types that represent base types from which all other composite types are constituted.
    • kTERM_STRUCTURES: Structures. Structures are nested collections of primitive data types. The data type refers to the structure, not to the individual primitives
    • kTERM_ENUMERATED: Enumerated. Enumerated type values must be selected among a finite set of values or from a controlled vocabulary. Attributes of this type are validated by the set domain.
    • kTERM_CARDINALITY: Cardinality. Cardinality types are used in conjunction with other types, they indicate whether the value is required or whether the value is a list. This type acts as a modifier for the other types.
    • kTERM_RELATIONSHIP: Relationship. Relationship types enumerate the different kinds of relation, they qualify the relationship flow.

Kind definitions.

This file contains the term global identifiers of the default node and term kind enumerations.

Ontology terms and nodes have a category called kind which represents the function of the term or node within the ontology:

  • kKIND_ROOT: Root. This represents a door or entry point of an ontology. It can be either the node from which the whole graph or tree originates from, or a node that represents a specific thematic entry point. In general, such objects will be used as elements of an index to an ontology.
  • kKIND_FEATURE: Feature . This kind of object defines a feature, property or attribute of an object that can be described or measured. This kind of node will generally be found as a leaf of the structure describing an object. Plant height is a plant characteristic that belongs to the category of morphological traits: the latter is not a feature, while plant height is.
  • kKIND_METHOD: Method node. This kind of node is required whenever an object's feature can be measured in different ways or with different workflows without becoming a different feature. Plant height is an attribute of a plant which can be measured after a month or at flowering time; the attribute is the same, but the method is different.
  • kKIND_SCALE: Scale node. This kind of node describes in what unit or scale a measurement is expressed in. Plant height may be measured in centimeters or inches, as well as in intervals or finite categories.
  • kKIND_INSTANCE: Instance node. In general, ontology nodes represent metadata, in some cases nodes may represent actual data: an instance node is a node that represents the metadata and data of an object. An ISO 3166 country code can be considered an enumeration node that constitutes the metadata for the country it represents, but if you store data regarding that country in the node, this may become also an instance node, because it represents the object it defines.
  • kKIND_ENUMERATION: Enumeration. This kind of node describes a controlled vocabulary element. These nodes derive from scale nodes and represent the valid choices of enumeration and enumerated set scale nodes. An ISO 3166 country code could be considered an enumeration node.
  • kKIND_NAMESPACE: Namespace. This kind of term represents a namespace, semantic group or ID space. Terms of this kind will generally be used to group identifiers.
  • kKIND_PREDICATE: Predicate. This kind of term represents a predicate, or element which connects subject and object vertices into a relationship.
  • kKIND_SYMMETRIC: Symmetric predicate. This kind of term represents a predicate which relates two terms in both directions. By default, relationships are directional, that is they originate from the subject towards the object, but not in the opposite direction: this kind of predicate, instead, does not have the concept of direction, the relationship flows in both directions.

Predicate definitions.

This file contains the term global identifiers of the default predicates.

Predicates are represented by a term object and serve as connectors between two vertex nodes, these are the default predicates:

  • kPREDICATE_SUBCLASS_OF: Subclass of. This predicate
    • indicates that the subject of the relationship is a subclass of the object of the relationship, in other words, the subject is derived from the object. This predicate is equivalent to the IS-A predicate, it states that the subject is an instance of the object.
  • kPREDICATE_SUBSET_OF: Subset of. This predicate indicates that the subject of the relationship represents a subset of the object of the relationship, in other words, the subject is a subset of the object, or the subject is contained by the object.
  • kPREDICATE_ATTRIBUTE_OF: Attribute of. This predicate indicates that the subject of the relationship is an attribute of the object of the relationship, this means that the subject of the relationship should be of the kKIND_FEATURE kind.
  • kPREDICATE_SCALE_OF: Scale of. This predicate is used by scale vertices to connect feature or trait vertices, the subject of the relationship represents a scale or unit and the predicate indicates that the object of the relationship uses that scale or unit. This predicate is specifically used to indicate the different scales in which a feature ir trait vertex can be expressed in.
  • kPREDICATE_METHOD_OF: Method of. This predicate relates method vertices with feature vertices or other methods, it indicates that the subject of the relationship is a method, or workflow variation of the object of the relationship. This predicate is used to connect the pipeline of modifiers applied to a feature vertex.
  • kPREDICATE_ENUM_OF: Enumeration of. This predicate relates vertex elements of an enumerated set, it indicates that the subject of the relationship is an enumerated set element instance, and if the object of the relationship is also an enumerated set element instance, this means that the subject is a subset of the object.
  • kPREDICATE_PREFERRED: Preferred choice. This predicate indicates that the object of the relationship is the preferred choice, in other words, if possible, one should use the object of the relationship in place of the subject. This predicate will be used in general by obsolete or deprecated items. The scope of this predicate is similar to the kPREDICATE_VALID predicate, except that in this case the use of the subject of the relationship is only deprecated, while in the kPREDICATE_VALID predicate it is not valid.
  • kPREDICATE_VALID: Valid choice. This predicate indicates that the object of the relationship is the valid choice, in other words, the subject of the relationship is obsolete or not valid, and one should use the object of the relationship in its place. This predicate will be used in general to store obsolete or deprecated versions. The scope of this predicate is similar to the kPREDICATE_PREFERRED predicate, except that in this case the use of the subject of the relationship is invalid, while in the kPREDICATE_PREFERRED predicate it is only deprecated.
  • kPREDICATE_LEGACY: Legacy version. This predicate indicates that the object of the relationship is the former or legacy choice, in other words, the object of the relationship is obsolete or not valid. This predicate will be used in general to record historical information. The scope of this predicate is similar to the kPREDICATE_PREFERRED and kPREDICATE_VALID predicates, except that in this case the legacy choice might not be invalid, nor deprecated: it only means that the object of the relationship was used in the past and the subject of the relationship is currently used in its place.
  • kPREDICATE_XREF: Cross reference. This predicate indicates that the subject of the relationship is related to the object of the relationship. This predicate will be found generally in both directions and does not represent any specific type of relationship, other than what the edge object attributes may indicate. The scope of this predicate is similar to the kPREDICATE_XREF_EXACT predicate, except that the latter indicates that the object of the relationship can be used in place of the subject, while in this predicate this is not necessarily true.
  • kPREDICATE_XREF_EXACT: Exact cross reference. This predicate indicates that the object of the relationship can be used in place of the subject of the relationship. If the predicate is found in both directions, one could say that the two vertices are identical, except for their formal representation. The scope of this predicate is similar to the XREF predicate, except that the latter only indicates that both vertices are related, this predicate indicates that they are interchangeable.

Type definitions.

This file contains the default data types.

Types are represented by term objects, they indicate the type of an object and are organised in the following enumerated set:

  • Primitive data types: These types represent primitive or atomic data types, which means that these cannot be decomposed in elementary components.
    • kTYPE_ANY: Any type. This type represents a type wildcard, it qualifies an attribute that can take any type of value.
    • kTYPE_STRING: String or text. This type defines a list of characters or a list of character strings.
    • kTYPE_INT: Integer number. This type defines an integer signed number, this number does not have a decimal part and has an indeterminate precision.
    • kTYPE_FLOAT: Floating point number. This type defines a signed floating point numeric value of indeterminate precision.

This abstract class is the ancestor of all connection and container classes in this library, it implements the the common interface shared by derived concrete instances.

A connection object holds a reference to a native connection which can be that to a server, database or container. The class server as a proxy to native implementations, offering a single interface to client objects which will use native derived concrete classes.

The Connection() accessor method can be used to manage the native connection and the inherited offset management methods can be used to manage the connection attributes.

Node object ancestor

A node is an element of a tree or graph which is connected to other nodes through edges, a node is a vertex of a relationship. This class concentrates in the core features of the node, without taking into consideration the connections to other nodes.

The class features the following default properties:

  • Term: The term, Term() or the kTAG_TERM tag, is a reference to a CTerm instance which holds the static attributes of the current node. This attribute is constituted by the kTAG_NID of the referenced term and is required.
  • Category: The category, Category() or the kTAG_CATEGORY tag, is an enumerated set of values that constitute the category or classification to which the object referred by the current node belongs to. This attribute is optional.
  • Kind: The kind, Kind() or the kTAG_KIND tag, is an enumerated set of values that define the function of the node. This attribute is optional.
  • Type: The type, Type() or the kTAG_TYPE tag, is an enumerated set of values that represent the type of the object defined by the current node. This attribute will be in general a combination of elements describing two main attribute categories: the data type and the cardinality. This attribute is optional.
  • Description: The description, Description() or the kTAG_DESCRIPTION tag, represents the description or long label of the node. This attribute is constituted by an array of elements in which the value of the element represents the description string and the element key is the language code in which the description is expressed in. This attribute is optional.

The node's main property is the term, kTAG_TERM, which defines the abstract concept that the node represents: a term taken by itself is an abstract concept, when it is referenced from a node which is related to other nodes through predicates, this term takes a different meaning, it becomes a term in context. Terms are like vocabulary, taken out of context they have a generalised meaning; put in context, they take a different meaning depending in what position of a term chain they are positioned. In this class, terms may hold any value.

The node also features a kTAG_KIND property which is an enumerated set describing the specific type, or the specific function of the node in its chain or path.

Finally, the class features the kTAG_TYPE property which defines the data type of the node or the data unit it represents, this offset is an enumerated set in which two elements can reside: the data type or unit, which is required, and optionally one or both kTYPE_REQUIRED and kTYPE_ARRAY which respectively indicate whether the node represents a required attribute and whether the data type indicated in the first element of the offset is a list or not.

This class does not feature attributes that can represent the unique identifier of the object, this means that the container is responsible of providing the primary key of the object.

Nodes can be considered _IsInited() when they have the kTAG_TERM offset set. This and the other properties can be changed at any time, none are locked once the object has been committed: it is the responsibility of the caller or of derived classes to implement a locking rule.

The class features member accessor methods for the default offsets:

  • Term(): This method manages the node's term, kTAG_TERM.
  • Kind(): This method manages the node's kind, kTAG_KIND.
  • Type(): This method manages the node's type, kTAG_TYPE.

COntology class definition.

This file contains the class definition of COntology which represents a high level ontology wrapper.

This class derives from CConnection and expects a CDatabase instance in its constructor. This class features methods that will create the default ontology, which is the set of elements with which the actual ontology is built.

The ontology is made of four main collections:

  • Terms: These elements are represented in this library by the instances of the COntologyTerm class, they represent concepts whose meaning is independent from their context. These objects may carry any number or type of attributes, the default ones which are managed by this library are the following:
    • kTAG_LID: Local identifier. This attribute contains the local unique identifier. This value represents the value that uniquely identifies an object within a specific domain or namespace. It is by definition a string constituting the suffix of the global identifier, kTAG_GID.
    • kTAG_GID: Global identifier. This attribute contains the global unique identifier. This value represents the value that uniquely identifies the object across domains and namespaces. This value is by definition a string and will generally constitute the object's primary key (kTAG_NID) in full or hashed form.
    • kTAG_NAMESPACE: Namespace. This attribute is a reference to a term which represents the namespace of the current object. The object local identifiers must be unique within the scope of the namespace.
    • kTAG_SYNONYMS: Synonyms. This attribute contains a list of strings that represent alternate identifiers for the hosting object. Synonyms do not have any relation to the namespace.
    • kTAG_TERM: Term. This attribute contains a reference to an object that represents the term of the attribute host.
    • kTAG_KIND: Kind. This attribute is an enumerated set that defines the kind of the hosting object.
    • kTAG_LABEL: Label. This attribute represents the label, name or short description of the referenced object. It is a kTYPE_LSTRING structure in which the label can be expressed in several languages.
    • kTAG_DEFINITION: Definition. This attribute represents the definition of the referenced object. It is an kTYPE_LSTRING structure in which the definition can be expressed in several languages. This attribute is similar to a description, except that the latter depends on the context, which is why it belongs to nodes, while the definition does not depend on the context, which is why it belongs to terms.
    • kTAG_NODES: Nodes. This attribute is a collection of node references, it is an array of node object native identifiers who reference the current object.
    • kTAG_FEATURES: Features. This attribute is a collection of feature references, it is an array of object native identifiers that reference the current object as a feature or trait.
    • kTAG_METHODS: Methods. This attribute is a collection of method references, it is an array of object native identifiers that reference the current object as a method or modifier.
    • kTAG_SCALES: Scales. This attribute is a collection of scale references, it is an array of object native identifiers that reference the current object as a scale or unit.
    • kTAG_NAMESPACE_REFS: Namespace references count. This integer attribute counts how many times external objects have referenced the current one as a namespace.
  • Nodes: These elements are represented in this library by the instances of the COntologyNode class, they represent concepts whose meaning is dependant from their context and require a reference to a term object. In a sense, nodes represent terms in context. These objects may carry any number or type of attributes, the default ones which are managed by this library are the following:
    • kTAG_TERM: Term. This attribute contains a reference to an object that represents the term of the attribute host.
    • kTAG_NODE: Node. This attribute contains a reference to an object that represents the node of the attribute host.
    • kTAG_CATEGORY: Category. This attribute is an enumerated set that contains all the categories to which the hosting object belongs to.
    • kTAG_KIND: Kind. This attribute is an enumerated set that defines the kind of the hosting object.
    • kTAG_TYPE: Type. This attribute is an enumerated set that contains a combination of data type and cardinality indicators which, combined, represet the data type of the hosting object.
    • kTAG_DESCRIPTION: Description. This attribute represents the description of the referenced object. It is an kTYPE_LSTRING structure in which the description can be expressed in several languages. This attribute is similar to a definition, except that the latter does not depend on the context, which is why it belongs to nodes, while the description depends on the context, which is why it belongs to nodes.
    • kTAG_NODES: Nodes. This attribute is a collection of node references, it is an array of node object native identifiers who reference the current object.
    • kTAG_EDGES: Edges. This attribute is a collection of edge references, it is an array of edge object native identifiers who reference the current object.
  • Edges: These elements are represented in this library by the instances of the COntologyEdge class, they represent relationships between node objects connected by a predicate term. They represent a subject/predicate/object triplet which connects all elements of the graph. These objects may carry any number or type of attributes, the default ones which are managed by this library are the following:
    • kTAG_GID: Global identifier. This attribute contains the global unique identifier. This value represents the value that uniquely identifies the object across domains and namespaces. This value is by definition a string and will generally constitute the object's primary key (kTAG_NID) in full or hashed form.
    • kTAG_UID: Unique identifier. This attribute contains the hashed unique identifier of an object in which its kTAG_NID is not related to the kTAG_GID. This is generally used when the kTAG_NID is a sequence number.
    • kTAG_SUBJECT: Subject. This attribute contains a reference to an object that represents the start, origin or subject vertex of a subject/predicate/object relationship.
    • kTAG_OBJECT: Object. This attribute contains a reference to an object that represents the end, destination or object vertex of a subject/predicate/object relationship.
    • kTAG_PREDICATE: Predicate. This attribute contains a reference to an object that represents the predicate term of a subject/predicate/object relationship.
  • Tags: These elements are represented in this library by the instances of the COntologyTag class, they represent paths between a starting node called feature, which represents the feature or trait to be tagged, and an ending node called scale which represents the scale or unit in which the feature is measured or described. Tags are the objects used to tie the ontology to data, these objects are literally used to tag datasets. These objects may carry any number or type of attributes, the default ones which are managed by this library are the following:
    • kTAG_GID: Global identifier. This attribute contains the global unique identifier. This value represents the value that uniquely identifies the object across domains and namespaces. This value is by definition a string and will generally constitute the object's primary key (kTAG_NID) in full or hashed form.
    • kTAG_UID: Unique identifier. This attribute contains the hashed unique identifier of an object in which its kTAG_NID is not related to the kTAG_GID. This is generally used when the kTAG_NID is a sequence number.
    • kTAG_PATH: Path. This attribute represents a sequence of vertex, predicate, vertex elements whose concatenation represents a path between an origin vertex and a destination vertex.

All four collections share the following two attributes:

  • kTAG_NID: Native identifier. This attribute contains the native unique identifier of the object. This identifier is used as the default unique key for all objects and can have any scalar data type.
  • kTAG_CLASS: Class. This attribute is a string that represets the referenced object's class. This attribute is used to instantiate the correct class once an object has been retrieved from its container.

This class extends its ancestor, CNode, by asserting the data types of the object's attributes and by setting related object's references.

  • Term: The term, or the inherited kTAG_TERM tag, in this class is a reference to an instance of the COntologyTerm class, constituted by its native identifier, kTAG_NID. At commit time, _PrecommitRelated(), we commit the eventual term instance and check if the value provided to the Term() member accessor is a term reference; we also manage its kTAG_NODES attribute by adding or removing the current node's kTAG_NID from the term's kTAG_NODES list depending on whether the term was referenced or the node deleted.
  • Type: The type, Type() or the inherited kTAG_TYPE tag, in this class is required to belong to a specific set of values, the Type() method's documentation lists them.
  • Edge references: The edge references, kTAG_EDGES tag, represents the COntologyEdge object references in which the current node was referenced as the subject or object of a relationship. It is either the count or an array of COntologyEdge kTAG_NID attributes, depending on the value of the kSWITCH_kTAG_EDGES flag:
    • 0x2: Keep count of edge references. This means that the kTAG_EDGES attribute will be a reference count.
    • 0x3: Keep list of edge references. This means that the kTAG_EDGES attribute will be a list of references.
    • 0x0 or other: Don't handle this information. This means that the kTAG_EDGES attribute will not be handled.
    This attribute is read-only and is managed internally.

Since the class represents a term in context, there is no combination of attributes that can be used to uniquely identify the object, for this reason the object does not feature the kTAG_GID attribute, so it is the duty of the hosting container to provide the kTAG_NID identifier. In this class we use two main strategies to identify nodes:

  • Sequence numbers: The kTAG_NID attribute is constituted by an integer which is a sequence number generated by the CContainer::NextSequence() method of a default container named kCONTAINER_SEQUENCE_NAME with a kSEQUENCE_KEY_NODE key.
  • Graph nodes: If the node container features the kOFFSET_GRAPH offset, this means that the server is also connected to a graph storage engine, in that case a new node will be created in the graph container, CGraphContainer, and the current object will use its identifier as the kTAG_NID attribute.

Once the node has been committed, it will not be possible to modify the term, kTAG_TERM.

When inserting a new node, the class will also make sure that the referenced term receives the newly created node identifier in its kTAG_NODES offset.

This class prevents updating the full object once it has been inserted for the first time. This behaviour is necessary because nodes are referenced by many other objects, so updating a full node object is risky, since it may have been updated elsewhere: for this reason the Update() and Replace() methods will raise an exception.

The class implements the static method, DefaultContainer(), which, given a database, will return the default nodes container; it will use the kCONTAINER_NODE_NAME constant as the container name. Note that when passing CConnection based objects to the persisting methods of this class, you should provide preferably database instances, since this class may have to commit terms.

This class extends its ancestor, CTag, by asserting the type of objects passed to the path, kTAG_PATH, and by using term references as the elements of the object's path and global identifier.

  • Path: As in the parent class, this attribute collects graph vertices and the predicates that connect them. The odd elements are expected to be vertex nodes of graph and the even elements the predicate terms that connect them. In this class the vertices are represented by the kTAG_TERM reference of the vertex COntologyNode instances, while the predicates are represented by the COntologyTerm kTAG_NID of the predicate terms.
  • Global identifier: The global identifier, GID(), in this class is represented by the concatenation of the kTAG_GID attributes of all the COntologyTerm instance references of the object's path, kTAG_PATH, separated by the kTOKEN_INDEX_SEPARATOR token.
  • Unique identifier: The unique identifier, kTAG_UID, in this class is represented by the hash of the global identifier.

Elements of the path can be provided as committed objects, they must have their kTAG_NID set. COntologyNode instances or references can only be provided as odd elements, while COntologyTerm instances or references can be provided as both odd or even elements. Note that nodes will always be resolved into the terms they refer to, so there there is no direct relationship between nodes and tags.

This particular behaviour allows for different graph paths to share the same tag, which allows duplicating ontologies for use as templates, or preventing duplicate data types from being tagged differently. It also allows the same vertex elements to be treated differently if their predicates differ.

Objects of this class will almost always be created by providing a sequence of node vertices and predicate terms, the PushItem() method will follow these rules: the first element of the path is the feature vertex, if provided as an COntologyNode instance, it must have its COntologyNode::Kind() set to kKIND_FEATURE; the last element of the path is the scale vertex, if provided as an COntologyNode instance, it must have its COntologyNode::Kind() set to kKIND_SCALE.

Persistent document ancestor

A persistent document object is a document object that knows how to persist in derived concrete instances of the CContainer class. This class features a default offset, kTAG_NID, which represents the database native unique identifier, or primary key, all objects derived from this class are uniquely identified by this offset.

The object can have any number of attributes, it is up to the user to decide what structure the object should have, these attributes are stored in the array part of the object and tagged by the array index.

This class implements a series of persistence methods:

  • Insert(): This method will insert the object into a container, if the object already exists, the method will raise an exception.
  • Update(): This method will update the object in a container, if the object does not exist, the method will raise an exception.
  • Replace(): This method will either insert, if the object is not found in the container, or update, if the object exists in the container.
  • Restore(): This method will load the object from the container and restore its data, the method expects the object to have its native unique identifier, kTAG_NID, set, or it will raise an exception.
  • Delete(): This method will delete the object from a container.
  • NewObject(): This static method can be used to instantiate an object by retrieving it from a container. The class of the object will be set according to the class used in the static call.

These methods take advantage of the inherited protected interface which can be used to process the object before, _Precommit(), and after, Postcommit() it is committed, these are the methods that derived classes should overload to implement custom workflows. In this class we use it to set the _IsCommitted() status and reset the _IsDirty() status whenever the object is stored or loaded from a container, and reset both when the object is deleted.

The method implements an accessor method for the native unique identifier, NID().

        • kOPERATOR_LESS_EQUAL: Smaller than or equal (<=).

        • kOPERATOR_GREAT_EQUAL: Greater than or equal (>=).

        • kTYPE_BINARY_STRING: A binary string.

This class manages a data member which can be used to track the state of the object.

Term object ancestor

A term is an object that features the following default properties:

  • Local identifier: The local identifier, LID() or the kTAG_LID tag, is a string code that uniquely identifies the object among all those sharing the same namespace. This attribute is required.

Wrapper

This class represents a web-services wrapper server, it is the ancestor of web-service classes in this library.

Wrappers are objects that respond to standard web calls and receive parameters in both GET and POST parameters, the main two parameters handled by this class are:

  • kAPI_FORMAT (required): The data format of the response,it will be returned as:
    • kTYPE_PHP: A PHP-serialised string.
    • kTYPE_JSON: A JSON-serialised string.
  • kAPI_OPERATION (required): The requested operation, each class will implement specialised handlers, this class only implements the following two operations:
    • kAPI_OP_HELP: A LIST-OP command, this command will return in the kAPI_RESPONSE section the list of supported operations as an array structured as follows:
      • index: The index will be the operation code.
      • value: The value will be the operation description.
    • kAPI_OP_PING: A PING command, this can be used to check if a service is alive.

If both the above parameters are present, the service will return an array constituted by the following three sections:

  • kAPI_STATUS: Operation status. This section is returned by default and will inform on the status of the requested operation. It consists of an array containing the following elements:
    • kTERM_STATUS_LEVEL: Response status. This element will be returned by default regardless of the operation outcome. This corresponds to the severity of the response and it can take the following values:
      • kSTATUS_IDLE: This is the status of the web-service before any operation has been executed, or when the operation was successful; this is the response of a successful kAPI_OP_PING request.
      • kSTATUS_NOTICE: The operation was successful and a notice message was returned.
      • kSTATUS_MESSAGE: The operation was successful and a message was returned.
      • kSTATUS_WARNING: The operation was successful but a warning was raised.
      • kSTATUS_ERROR: The operation failed because of an error.
      • kSTATUS_FATAL: The operation failed because of a fatal error, this will generally mean that the web-service is not operational.
      • kSTATUS_BUG: The operation failed because of a bug, the developers should be informed of this kind of errors.
    • kTERM_STATUS_CODE: Status code. This element will be returned by default regardless of the operation outcome. It corresponds to the error code; kERROR_OK means no error.
    • kTERM_STATUS_MESSAGE: Status message. The response message from the operation, this element is used to return informative messages or to return error messages when the service fails. It will generally be formatted as an array in which the index represents the language code in which the message, stored in the element data, is expressed in.
    • kAPI_AFFECTED_COUNT: Record count. The total number of elements affected by the operation. This tag will only be used by derived classes returning data elements.
    • kTERM_NOTES: Attachments. A list of key/value pairs containing information relevant to the operation response. For instance, if a series of parameters are required and were not provided, this could list them.
  • kAPI_REQUEST: Service request. This section will return the actual request provided to the service, this can be used for debugging purposes and will only occur if the optional kAPI_LOG_REQUEST parameter has been set to 1.
  • kAPI_STAMP: Timers. This section holds timing information, it will be returned only if you provide the time of day [gettimeofday( TRUE )] in the kAPI_STAMP_REQUEST parameter. This section is structured as follows:
    • kAPI_STAMP_REQUEST: Request time stamp, the time in which the request was sent; this is the same value sent by the caller in the kAPI_STAMP_REQUEST parameter.
    • kAPI_STAMP_PARSE: Parse time stamp, the time in which the service finished parsing the request.
    • kAPI_STAMP_SENT: Response time stamp, the time in which the response was sent.
  • kAPI_RESPONSE: Response, this section will hold the operation response, in this class we only respond to kAPI_OP_HELP list requests.

Besides the kAPI_FORMAT and kAPI_OPERATION parameters described in the first section, we have three other optional parameters that can be used to receive specific information sections in the response:

  • kAPI_LOG_REQUEST: Log the request, if the value of this parameter is 1, the response will contain the received request in the kAPI_REQUEST section.
  • kAPI_LOG_TRACE: Trace exceptions, if the value of this parameter is 1, in the case of an error that triggered an exception, the error response will also include the call trace.
  • kAPI_STAMP_REQUEST: This parameter should hold the timestamp [gettimeofday( TRUE )] in which the client has sent the request, if provided, the service will return the timing information in the kAPI_STAMP section.

The parameters are expected either in GET or POST.

If either the kAPI_FORMAT or the kAPI_OPERATION parameters are omitted from the request, the service will return an empty response; this is to prevent unnecessary traffic.

Instances of this class can be considered server objects, and can be implemented with this simple code snippet:

$server = new CWrapper( new Server() ); $server->HandleRequest();

An example of this class implementation can be found in the Wrapper.php source file.
