Update related objects after committing

In this class we handle referenced master nodes, kTAG_NODE: depending on the value of the kSWITCH_kTAG_ALIAS_NODES flag we do the following:

  • 0x2: Keep count of references. This means that the kTAG_NODES attribute of the master node referenced by the kTAG_NODE attribute will be incremented when the object is inserted, or decremented when deleted.
  • 0x3: Keep list of references. This means that a reference to the current object will be added to the kTAG_NODES attribute of the master node referenced by kTAG_NODE attribute of the current object when the latter is inserted, and removed when the object isdeleted.
  • 0x0 or other: Don't handle this information. This means that the kTAG_NODES attribute will not be handled.

Handle embedded or related objects before committing

In this class we handle the master node: we check whether a master node related to the same term exists, if that is the case we set the current node's kTAG_NODE attribute, if that is not the case, we instantiate and insert the master node before setting its identifier in the current object's kTAG_NODE attribute.

Note that we do the above only if the kTAG_NODE offset is not yet set.

Manage node reference

This method can be used to manage the master node reference, kTAG_NODE, the method accepts a parameter which represents either the node, or the requested operation, depending on its value:

  • NULL: Return the current value.
  • FALSE: Delete the current value.
  • other: Set the value with the provided parameter:
    • integer: This value will be interpreted as the node reference.
    • COntologyNode: This value will be interpreted as the node object.
    • other: Any other value will raise an exception.

The second parameter is a boolean which if TRUE will return the old value when replacing containers; if FALSE, it will return the currently set value.

Relate to node

We overload this method to mirror relationships between alias nodes to their counterpart master nodes, this is governed by the $doPropagate switch.

We first create the requested relationship, then we resolve both the subject and the object, if it is also an alias, and we relate them with the same predicate.

This means that:

  • ALIAS => MASTER:
    • ALIAS => MASTER
    • ALIAS.MASTER => MASTER
  • ALIAS => ALIAS:
    • ALIAS => ALIAS
    • ALIAS.MASTER => ALIAS.MASTER

We require the connection parameter here since both subject and object vertices must be committed by this method, this implies that the edge will also be committed.
